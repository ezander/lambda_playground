#* 
  Assertions
  ==========

  To assert whether two λ expressions are αβ-equivalent (i.e. after doing all
  possible β reductions (=substitutions) the resulting expressions are
  α-equivalent, i.e. equal except for some renaming of *bound* variables), you
  can use the assertions operators `≡` and `≢` (type '\equiv' and '\nequiv' or
  Alt-e/Alt-n). 

    ≡ <lambda-expr1> <lambda-expr2>

  asserts that both expressions are (more or less, see later paragraph)
  αβ-equivalent, while 

    ≢ <lambda-expr1> <lambda-expr2>

  asserts that they are not. If such an assertion fails program evaluation 
  stops after this line. 
    
  Note, that "≡ expr1 expr2" means if it passes, that we could reduce both sides
  to some α-equivalent normal forms. If it's fails *and* both sides could be
  reduced to normal form, it also means, they are not equivalent. However, if
  it's false and they weren't reduced to normal form, it's not a proof that both
  sides are not equivalent - they still could be. It's maybe just that the
  reductions needed exceeded the step limit or the intermediate results exceeded
  the size limit.
  
  In the same spirit `≢` means that within the given computational limits both
  sides could not be reduced to α-equivent normal forms. If it succeeds it could
  still be that both sides have the same normal form, which only could not be 
  reached within limits. Or maybe that both sides have no normal forms, which 
  is why e.g. `≢ (ω ω) (ω ω)` succeeds for `ω := λx . x x`.

  Assertions work also with comprehensions. As rationale look e.g. at booleans:
  you want to assert that with a certain definition of boolean some standard 
  equalities like "not (not p) = p" or "not(p and q) = not p or (not q)" hold.
  However, asserting them directly with free variables p and q does not work,
  because they could be general λ expression, not restricted to booleans (that's
  a small downside of the untyped LC - you could plug in everything here). To
  overcome this, you can "bind" those free variables in the assertion in order 
  to allow only certain values for p and q. E.g. instead of writing 
  
    ≡ (not (and p q)) (or (not p) (not q))

  which would fail, you can write

    ≡[p:={true,false},q:={true,false}] (not (and p q)) (or (not p) (not q))

  so that the assertions is evaluated for all combinations of true and false for
  p and q.

  Some examples:
*#

# Assertions with non-normalizable forms never succeed
ω := λx . x x
≢ (ω ω) (ω ω)

#! include "sys/Church Booleans"

# Assertions that work without comprehensions
≡ true (not false)

# even with free variables `then` and `else`
≡ (if true then else) then
≡ (if false then else) else

# Assertions with (free) variables that need comprehensions to work, because 
# they would not hold for arbitrary lambda expressions

# Double negation: test that ¬¬p = p 
≢ (not (not p)) p
≡[p:={true,false}] (not (not p)) p

# Implication definition: test that (p→q) = ¬p ∨ q
≢ (or (not p) q) (implies p q)
≡[p:={true,false},q:={true,false}] (or (not p) q) (implies p q)

# Biconditional (logical equivalence, iff) definition: 
#    test that (p↔q) = (p→q) ∨ (q→p)
≢ (and (implies p q) (implies q p)) (iff p q)
≡[p:={true,false},q:={true,false}] (and (implies p q) (implies q p)) (iff p q)
