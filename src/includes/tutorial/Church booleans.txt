#*
  Church booleans
  ===============
  
  One of the most fundamental concepts in any programming language is that of a 
  boolean. No constrol structure (if-else, while, etc.) can be defined with a
  notion of booleans. So, how do we define booleans in a world where there's 
  only functions?

  It may sound somewhat philosophical to ask "What exacatly *is* or constitutes
  a boolean value - true or false". And in contrast maybe to classical
  philosophy the answer in computing is: it doesn't really matter. All that
  matters is behavior. But `true`` and `false` don't show behavior by
  themselves, only in connection with some selector function, that generally
  called `if` or `if-then-else`, with the expected behavior that `if true a b`
  evaluates to `a`, whatever the term `a` may be, and that `if false a b`
  evaluates to `b`. How we define `true`, `false` and `if` is not important as
  long the given conditions are fulfilled.

  In λ calculus there are multiple ways to do this (we'll see later), but the
  most common (nearly canonical) choice are the so-called Church booleans. Their
  definition is given by:
*#

true  := λ x y . x
false := λ x y . y

#* 
  i.e. `true`` and `false` are each functions of two arguments that return their
  first and second argument respectively. How do we define `if` now? `if` is
  obviously a function of three arguments, taking such a bool as it's first one
  and two unspecified one as second and third, i.e. something like `if := λ p a
  b . ???` as `p` is a boolean predicate, i.e. either `true` or `false`. Assume,
  `p` is `true`: if we apply `a` and `b` to `true`, we get `a` (true a b ≡
  (x)[x:=a] ≡ a). Assume, `p` is `false`: if we apply `a` and `b` to `false`, we
  get `b` (false a b ≡ (x)[x:=a] ≡ a).

  I.e. we can just define
*#

if := λ p a b . p a b
